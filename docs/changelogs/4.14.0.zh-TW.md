# Bee.NET 4.14.0 — 詳細發佈說明

[English](4.14.0.md) ・ [繁體中文](4.14.0.zh-TW.md) ・ [← CHANGELOG](../../CHANGELOG.zh-TW.md)

本檔是 [CHANGELOG.zh-TW.md](../../CHANGELOG.zh-TW.md) 中 `[4.14.0]` 條目的詳細版。主 changelog 每項一行；本檔保留完整敘事 —— 每項變更的 *為什麼*、限制與設計脈絡。

> Bee.NET 仍處 pre-stable 演進階段。本版落地兩大子系統與一項橫切正規化。**運算式與規則引擎**把欄位運算與驗證從手寫 BO 移到宣告式 `FormSchema` 定義 —— 計算欄即 `ValueExpression`、驗證即 `FormRule`，同一份可攜引擎在後端（存檔前、權威）與 Avalonia 前端（即時預覽）共用。**稽核軌跡子系統**將六條獨立日誌軸寫入 `st_log_*` 表，並提供查詢讀取側。最後，記憶體 `DataSet` 欄名正規化為**小寫**，使欄名在每一層 —— 定義、資料庫、DataSet、運算式、UI —— 寫法一致，消除接運算式引擎時浮現的大小寫敏感名稱比對問題。最後這項會改變 wire、屬破壞性；依 pre-stable 政策以 minor 發佈。設計理由見 [ADR-028](../adr/adr-028-expression-rule-engine.md)、[ADR-027](../adr/adr-027-audit-trail.md)、[ADR-029](../adr/adr-029-lowercase-field-names.md)。

## 破壞性變更

- **`DataSet` 欄名改小寫（wire 與公開 API）** —— 框架長年將記憶體 `DataSet` 欄名存為大寫，這是多年前為配合某個區分大小寫的 UI 繫結路徑而做的正規化。該大小寫洩漏到 wire，故 JSON / MessagePack payload 帶大寫 key 如 `SYS_ROWID`。自 4.14.0 起，各層一律以小寫為標準：`DataTableExtensions.AddColumn` 轉小寫（invariant，並修一個潛在的 Turkish-I 隱患），`DbAccess` 讀取邊界呼叫 `LowercaseColumnNames`（由 `UppercaseColumnNames` 更名），把各 provider 原生大小寫（Oracle 回大寫、PostgreSQL 回小寫）折成單一小寫。JSON / MessagePack converter 直出 `DataColumn.ColumnName`，故 payload key 現為小寫（`sys_rowid`）。**.NET 消費端不受影響**：`DataColumnCollection` 查找、`GetFieldValue`、運算式引擎皆以大小寫無關比對名稱（引擎以 `FormField.FieldName` 綁定）。僅兩類需處理 —— 以字面大寫 key 讀 `DataSet` 的外部 JS/TS client、以及呼叫已更名的 `UppercaseColumnNames` 的程式。完整理由與多 DB 驗證見 [ADR-029](../adr/adr-029-lowercase-field-names.md) 與遷移 plan。

## 新增

- **運算式與規則引擎（`Bee.Expressions`，新套件）** —— 可攜、沙箱化的求值引擎（`IExpressionEvaluator` / `DynamicExpressoEvaluator`，DynamicExpresso 封裝，MIT）。運算式為 C# 子集（`unit_price * quantity`、`amount > 0`、`status == "Approved"`）；沙箱只曝露欄位變數、精選 helper（`Today` / `Now` / `IsNullOrEmpty` / `Guid`）與基本型別 —— 其餘識別字於 parse 期即報錯。每條運算式編譯一次、以文字 + 參數簽章快取。`ExpressionPolicy` 集中型別/null 政策（`FieldDbType` → CLR、`DBNull` → 型別預設值，使 `unit_price * qty` 遇空值算 0），確保前端預覽與後端存檔結果一致。引擎亦提供引用變數分析，驅動「改哪個欄要重算哪個欄」的相依圖。[ADR-028](../adr/adr-028-expression-rule-engine.md)

- **schema 上的欄位運算式與規則（`Bee.Definition`）** —— `FormField` 新增 `ValueExpression`（計算欄，存檔前重算並捨入）與 `DefaultValueExpression`（新列以運算式產生的預設值）。`FormSchema` 新增 `FormRule` / `FormRuleCollection`：每條規則有選填的 `When` 適用性 guard、必須成立的 `Condition`、`Message`（字面或語系資源 key）、`Trigger`（`BeforeSave` | `BeforeDelete`）、選填 `TargetTable`、`Enabled` 與 `Order`。共用的 `FormExpressionCalculator`（於 `Bee.Definition.Forms`，鏡射 `FormRowDefaults`）承載 row-level 運算/驗證邏輯，使前後端走同一路徑、不漂移。

- **BO 生命週期模板方法（`Bee.Business`）** —— `FormBusinessObject.Save` / `Delete` 重構為模板方法：授權、記錄範圍、變更稽核固定於編排層，`DoBeforeSave` / `DoSave` / `DoAfterSave`（及刪除對應）為 `protected virtual` 擴充點。基底 `DoBeforeSave` 經 `IFormRuleProcessor` 依序套用 schema 驅動的預設值運算式、計算欄（全精度後經 `NumberFormatResolver.RoundByKind` 捨入）、`BeforeSave` 驗證規則。一般 CRUD 表單**零 BO 程式碼**；自訂 BO 覆寫某 `Do*` 並先呼叫 `base`。Northwind `OrderBO` 作為示範遷移：明細金額改 `ValueExpression`、必填檢查改 `FormRule`，只有真正命令式的部分（單號序列、存量狀態 guard）留在程式碼。

- **前端即時預覽（`Bee.UI.Avalonia`）** —— `FormLiveComputation` 於編輯時重算計算欄，訂閱統一的 `FormDataObject.FieldValueChanged`。master 編輯器透過自身繫結刷新；detail grid 的 realized cell 以 dispatcher-post 的 `RefreshRows` 刷新。相依圖 gating 重算、re-entrancy guard 防止回寫再觸發、任何求值失敗即降級為 no-op（後端仍權威）。Tier 2 捨入讓預覽對齊後端位數：`FormView` 由 `ClientInfo.Company` 加幣別/單位 master 組出 `RoundingContext`，注入計算器與金額/數量編輯器。AOT 已於 reflection-only 路徑（`IsDynamicCodeSupported=false`）驗證，故引擎在 WASM / 行動端可行、無需停用即時預覽。

- **稽核軌跡子系統（`Bee.Business` / `Bee.Repository`）** —— 六條獨立日誌軸寫入 `log` 分類的 `st_log_*` 表（`login`、`change`、`access`、`anomaly_api`、`anomaly_db`）。`IAuditLogWriter` 以背景 channel 批次寫入，埋點不阻塞請求。異動軸於存檔/刪除擷取 `DataSet` DiffGram（同時含現值與原值），提供完整前後影像。六軸設計與 `st_log_*` 命名見 [ADR-027](../adr/adr-027-audit-trail.md)。

- **稽核日誌查詢讀取側（`Bee.Business` / `Bee.Api.*`）** —— `GetChangeLog` / `GetChangeDetail` 驅動異動軸二段式瀏覽（先每筆記錄清單、再單筆的欄位級 DiffGram 明細）；平行清單查詢涵蓋登入、檢視、異常軸；異常彙總回傳 `Summary` 加 Top-N。中途引入的冗餘 `GetRecordHistory` 於 `GetChangeLog` + `GetChangeDetail` 涵蓋後移除，未曾發佈。

- **前端權限 capability（`Bee.UI.Avalonia` / `Bee.UI.Core`）** —— `EnterCompany` capability 快照現於前端降級 UI 元件：無 Read 的欄位隱藏、無 Update 的欄位唯讀（純 UX 降級；後端仍為權威邊界）。`ClientInfo` 快取進入的 `CompanyInfo`（`ClientInfo.Company`），`ClientDefineAccess` 新增 `GetCurrencySettingsAsync` / `GetUnitSettingsAsync`。record scope 權限亦支援多個 Owner / Dept 欄，以 OR 聯集求值。

## 變更

- **SQL Server `DateTime` → `datetime2(7)`（`Bee.Db`）** —— SQL Server `DateTime` 欄改用 `datetime2(7)` 取代 `datetime`，取得亞毫秒精度與 pre-1753 日期範圍。`datetime2` 參數改寫僅限 SQL Server，故 PostgreSQL / Oracle seeding 不受影響（中途的相關回歸已修）。

- **字串 key 比較 → `OrdinalIgnoreCase`（`Bee.Base`）** —— 對字串 key（集合 key、欄名、id）的大小寫無關比較統一收斂為 `OrdinalIgnoreCase`，取代文化相依比較，使其文化無關且最快，並避開 `CurrentCultureIgnoreCase` 在土耳其地區設定下把 `ID` 判為 ≠ `id` 的 Turkish-I 隱患。

## 修正

- **運算式變數綁定與值 coercion（`Bee.Expressions`）** —— 三個相關修正，於 Northwind 訂單表單 dogfooding 引擎時浮現：
  - 變數表現以宣告的 `FormField.FieldName` 為 key，而非原始 `DataColumn.ColumnName`。因 `AddColumn` 把欄名存大寫、DynamicExpresso 識別字區分大小寫，運算式引用宣告的小寫 `quantity` 無法對應 `QUANTITY` 欄（未知識別字錯誤，存檔時呈現為 `-32000` 內部錯誤）。以宣告欄名為 key 使運算式識別字與 DataSet 儲存大小寫解耦。（4.14.0 的小寫正規化亦從根源消除此落差。）
  - `ExpressionPolicy.CoerceValue` 現處理 string 型 `Guid` 與 `byte[]` 欄 —— wire 反序列化或 SQLite 來源的欄常以 string 形式抵達（SQLite 以 TEXT 儲存 GUID）。`Convert.ChangeType` 對這些非 `IConvertible` 型別會拋例外；coercion 改為顯式解析。
  - 空字串 GUID（未選的 key、TEXT 型別空 GUID 欄）對映為 `Guid.Empty`，而非於 `Guid.Parse("")` 拋例外，對齊 null/DBNull 預設值政策。
