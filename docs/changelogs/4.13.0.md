# Bee.NET 4.13.0 — Detailed Release Notes

[English](4.13.0.md) ・ [繁體中文](4.13.0.zh-TW.md) ・ [← CHANGELOG](../../CHANGELOG.md)

This file is the detailed companion to the `[4.13.0]` entry in [CHANGELOG.md](../../CHANGELOG.md). The top-level changelog lists the headline items in one line each; this file keeps the full narrative — the *why*, the constraints, and the design context behind each change.

> Bee.NET remains in pre-stable evolution. This release adds an ERP-grade numeric layer built around a single idea: a numeric field carries a semantic **`NumberKind`**, and the kind — not the field's storage type — decides how the value is displayed, whether it is rounded on write, and where its decimal places come from. On top of that sit two ERP invariants (**round-then-sum** and **preserve-never-rounds**) and two runtime-resolved decimal sources (**currency** and **unit of measure**), mirroring SAP's CURR/CUKY and QUAN/UNIT families. Every addition is backward compatible — new members carry `[DefaultValue]` empty, `CompanyInfo`'s new MessagePack keys are tail-appended, and missing masters fall back to framework defaults — so existing schemas, XML, and wire payloads are unaffected. The full design rationale and the SAP / Odoo comparison live in [ADR-026](../adr/adr-026-numeric-semantics-rounding.md).

### Added

- **`NumberKind` semantic on fields (`Bee.Definition`)** — `FormField` (propagated to `LayoutFieldBase`) now carries a `NumberKind`: `Quantity`, `Weight`, `Amount`, `Percent`, `UnitPrice`, `Cost`, or `ExchangeRate`. The kind is the single point of truth for three things — the display format letter (`N` / `P`), the rounding policy (`Round` vs `Preserve`), and the decimals source (currency / unit / company / system-fixed). Framework defaults: `Quantity` 0, `Weight` 3, `Amount` 2, `Percent` 2, `UnitPrice` / `Cost` 4, `ExchangeRate` 5. `NumberKindProfile` exposes the profile; it replaces the isolated, zero-caller `NumberFormatPresets`. See [ADR-026](../adr/adr-026-numeric-semantics-rounding.md) for why the semantic lives on the field rather than being inferred from the DB type.

- **Rounding and formatting engine (`Bee.Definition`)** — `NumberFormatResolver` provides `ResolveDecimals` / `ResolveFormat` / `RoundByKind` / `RoundCash`, and `NumberFormatApplier.Bake` stamps display formats onto a schema. Two ERP invariants are enforced here:
  - **Round-then-sum.** For `Round` kinds a total equals the *sum of already-rounded details*, never a full-precision sum rounded once at the end. `RoundByKind` rounds each detail (AwayFromZero) to its resolved decimals; callers add the rounded values, so `Σ details == total` by construction. Original and home amounts round to their own currency independently.
  - **Two-layer rounding.** The detail layer rounds to the *natural* currency / unit decimals (`CurrencySettings` / `UnitSettings`); an optional final **cash rounding** (`RoundCash`, SAP T001R-style, e.g. CHF → 0.05) snaps only the final payable and books the deliberate `payable − total` difference to a rounding account. Currency decimals are system-wide; only the cash-rounding unit is company-overridable.
  - **Preserve never writes a rounded value.** `UnitPrice` / `Cost` / `ExchangeRate` pass through `RoundByKind` unchanged — their decimals are display-only, and rounding a source value would inject error downstream.
  - Baking runs on a **per-call clone** inside `SystemBusinessObject.LoadAndLocalizeSchema`; the process-shared cache is never mutated, and an author-supplied `NumberFormat` always wins.

- **Multi-currency: `CurrencySettings` + per-field CUKY binding (`Bee.Definition`)** — a new system-level currency master (`DefineType.CurrencySettings`, curated ISO 4217, SAP TCURX-style) ships a `CurrencyItem` per code (`Code` / `Numeric` / `Rounding` / `Symbol` / `Name`), from which the natural decimals are derived (JPY = 0, USD = 2, BHD = 3). Amount fields resolve their decimals **at runtime by their currency**, not the company: each amount binds a currency-key field via `FormField.CurrencyField`, the master document currency lives on `FormSchema.CurrencyField` (by convention `sys_currency`), and the resolution priority is explicit `CurrencyField` → master `sys_currency` → company `DefaultCurrency` → framework 2. Because currency is document data that changes, `Bake` deliberately does *not* bake amount formats — it stamps the effective currency-reference field so the UI can resolve per row. The master ships to the client through the ordinary `GetDefine` channel; a missing master falls back to 2.

- **Unit of measure: `UnitSettings` + per-field UNIT binding (`Bee.Definition`)** — exactly parallel to currency (SAP QUAN/UNIT mirrors CURR/CUKY). A system-level unit master (`DefineType.UnitSettings`, SAP T006-style) carries a `UnitItem` per code (`Code` / `Decimals` / `Dimension` / `Name`) storing its decimals directly (KG = 3, PCS = 0). `Quantity` / `Weight` fields bind a unit field via `FormField.UnitField` (units are per row — there is no master-level unit); resolution priority is bound `UnitField` value → company decimals → framework default. Round-then-sum holds per unit; a mixed-unit column has no meaningful total.

- **Company-level numeric model (`Bee.Definition`)** — `CompanyInfo` gains four tail-appended members: `NumberFormats` (`[Key(4)]`, a `CompanyNumberFormats` override table for `Percent` / `UnitPrice` / `Cost` and the quantity/weight no-unit fallback), `DefaultCurrency` (`[Key(5)]`, the company home currency), `CashRounding` (`[Key(6)]`, a `CompanyCashRounding` per-currency cash-rounding table), and `AllowedCurrencies` (`[Key(7)]`, a per-company currency whitelist bounding a document's currency dropdown). These are backed by four new `st_company` columns (`number_formats_xml`, `default_currency`, `cash_rounding_xml`, `allowed_currencies_xml`), read by `CompanyRepository.GetById`. Because the keys are tail-appended, existing MessagePack payloads stay compatible; empty columns fall back to framework defaults / empty whitelist (= all currencies).

- **`NumericEdit` editor (`Bee.UI.Avalonia`)** — a new `ControlType.NumericEdit` (auto-resolved by `LayoutColumnFactory` / `FieldEditorFactory` for `Short` / `Integer` / `Long` / `Decimal` / `Currency`). It shows the full stored precision on focus, formats per the field's `NumberFormat` on blur, right-aligns, and keeps invalid input as the prior value. Crucially, the display rounding is never written back — `GetWriteBackValue` parses and returns the full-precision value.

- **Currency / unit-aware grid formatting (`Bee.UI.Avalonia`)** — `GridControl` resolves each amount / quantity cell's format **per row** by reading that row's `CurrencyField` / `UnitField` value (falling back to a default code), so a single column renders USD and JPY rows at their own decimals; leaving the settings null preserves the earlier column-level baked behavior. `AmountColumnSummary.TryComputeTotal` is a footer helper that totals a same-currency (or same-unit) column and returns null for a mixed column — matching Odoo's monetary-sum behavior where a mixed total is meaningless. Both `NumericEdit` and the grid share the same per-cell resolution path.

### Notes

- **No breaking changes.** New `FormField` / `LayoutFieldBase` / `FormSchema` members default to empty and do not alter existing XML round-trips; `CompanyInfo`'s new keys are tail-appended for MessagePack wire compatibility; the new `DefineType` members fall back when no master is defined; `NumberFormatPresets` (zero production callers) was refactored to the `NumberKind` model with only its own tests adjusted.
- **`Bee.UI.Avalonia` is the pilot.** The `NumericEdit` editor and grid formatting land in the Avalonia control family first; the MAUI / Blazor ports follow separately.
