# Bee.NET 4.15.0 — Detailed Release Notes

[English](4.15.0.md) ・ [繁體中文](4.15.0.zh-TW.md) ・ [← CHANGELOG](../../CHANGELOG.md)

This file is the detailed companion to the `[4.15.0]` entry in [CHANGELOG.md](../../CHANGELOG.md). The top-level changelog lists the headline items in one line each; this file keeps the full narrative — the *why*, the constraints, and the design context behind each change.

> Bee.NET remains in pre-stable evolution. This release carries no new features; it is a deliberate **wire & API consolidation** to lock the serialization and contract-surface conventions before stabilization. Two changes stand out. First, MessagePack contract serialization moves from positional integer keys to **property-name keys**, so a single name-based convention now governs both JSON and MessagePack payloads and the historical footguns of integer-keyed layouts disappear. Second, the API **contract interfaces are reorganized into per-axis namespaces** so the interface layer finally matches the axis split that the implementation and message layers already use. Both are technically breaking, but with no external consumers yet they ship as a minor under the pre-stable policy. Design rationale for the serialization change lives in [ADR-030](../adr/adr-030-messagepack-name-based-keys.md).

## Breaking changes

- **MessagePack contracts use property-name keys (wire)** — the framework historically annotated contract DTOs with positional integer `[Key(n)]` attributes, which serialize to a compact MessagePack array. That layout has two long-standing hazards: the constructor-parameter order of a collection item must be kept in lock-step with its `[Key]` order (a silent field-swap on the wire if they drift), and key numbers have to be coordinated by hand across an inheritance chain. As of 4.15.0, 72 contract types switch to `[MessagePackObject(keyAsPropertyName: true)]` and drop their integer keys: 57 `Bee.Api.Core.Messages` request/response types (including `ApiMessageBase.Parameters`, whose `[Key(0)]` is removed) and 15 `Bee.Definition` / `Bee.Api.Contracts` DTOs and non-`[Union]` collection items (`PackageUpdateInfo` / `PackageUpdateQuery`, `RecordFieldChange`, `CompanyInfo`, `DepartmentTree` / `DepartmentNode`, `PagingInfo` / `PagingOptions`, `AllowedCurrencyItem`, `CashRoundingItem`, `NumberFormatItem`, `CurrencyItem`, `UnitItem`, `SortField`, `Parameter`). The MessagePack payload therefore changes from a positional array to a property-name map, structurally matching the JSON payload. This is a breaking wire change; because client and server ship together and there are no external consumers, there is no compatibility cost today, but a client and server must run compatible versions — the new name-based formatters cannot read old positional-key payloads. The migration was **category-aware**, not a blanket conversion: `[Union]` polymorphic types (`FilterNode` / `FilterCondition` / `FilterGroup`) stay on integer keys because `[Union]` discrimination is incompatible with `keyAsPropertyName`; collection containers (`MessagePackCollectionBase` / `KeyCollectionBase` subclasses) go through their own `CollectionBaseFormatter` / `ItemsForSerialization` proxy and are unaffected by the key strategy; and `SerializableData*` (DataSet/DataTable wire plumbing) keeps integer keys. Phase 0 confirmed that `keyAsPropertyName` also round-trips correctly on the reflection-only (AOT) path, so MessagePack's source generator is not a prerequisite. [ADR-030](../adr/adr-030-messagepack-name-based-keys.md)

- **Contract interfaces reorganized into per-axis namespaces (source)** — the `Bee.Api.Contracts` package historically kept every contract interface flat in the root namespace, even though the corresponding implementation (`Bee.Business.System` / `.Form` / `.AuditLog`) and message (`Bee.Api.Core.Messages.System` / `.Form` / `.AuditLog`) types were already split by axis. That left the interface layer as the only place where "the contract implementation is in an axis folder but the contract interface sits flat" was still true. In 4.15.0 the interfaces move into matching subfolders / namespaces: the System axis (29 interfaces + the `PackageDelivery` / `PackageUpdateInfo` / `PackageUpdateQuery` DTOs) into `Bee.Api.Contracts.System`, the Form axis (12 interfaces: `IDelete` / `IGetData` / `IGetList` / `IGetLookup` / `IGetNewData` / `ISave`, each Request/Response) into `Bee.Api.Contracts.Form`, and the AuditLog axis (13 interfaces + the `RecordFieldChange` DTO) into `Bee.Api.Contracts.AuditLog`. The root `Bee.Api.Contracts` namespace now holds only the cross-BO, BO-agnostic `ExecFunc` contracts (`IExecFuncRequest` / `IExecFuncResponse`), mirroring the root-level `ExecFunc` implementation in the message layer. This is a pure source-level change: the serialization implementation classes' namespaces are unchanged, so there is no wire impact. External consumers that reference the moved interfaces update their `using` directives (see below).

## Changed

- See the two breaking-change entries above — this release's user-facing surface is entirely the serialization and contract-namespace consolidation.

## Fixed

- Nothing user-facing. (Internally, a sample factory that was missing the `CreateLogBusinessObject` override added in 4.14.0 was fixed, and a newly-published AngleSharp advisory was patched in the Blazor test projects — neither affects the shipped packages.)

## Upgrade

External consumers referencing the moved contract interfaces update their `using` directives to the per-axis namespace:

```diff
- using Bee.Api.Contracts;
+ using Bee.Api.Contracts.System;   // ILoginRequest, IPingRequest, …
+ using Bee.Api.Contracts.Form;     // IGetListRequest, ISaveRequest, …
+ using Bee.Api.Contracts.AuditLog; // change-axis contracts, RecordFieldChange
```

The `ExecFunc` contracts (`IExecFuncRequest` / `IExecFuncResponse`) remain in the root `Bee.Api.Contracts` namespace and need no change.

The MessagePack wire-format change requires no code change, but client and server must run the same (or a compatible) version — payloads written with the old positional integer keys cannot be read by the new name-based formatters.
