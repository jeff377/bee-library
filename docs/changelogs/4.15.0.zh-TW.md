# Bee.NET 4.15.0 — 詳細發佈說明

[English](4.15.0.md) ・ [繁體中文](4.15.0.zh-TW.md) ・ [← CHANGELOG](../../CHANGELOG.zh-TW.md)

本檔是 [CHANGELOG.zh-TW.md](../../CHANGELOG.zh-TW.md) 中 `[4.15.0]` 條目的詳細版。主 changelog 每項一行;本檔保留完整敘事 —— 每項變更的 *為什麼*、限制與設計脈絡。

> Bee.NET 仍處 pre-stable 演進階段。本版無新功能,是發版前刻意的 **wire 與 API 收斂**,在穩定化之前把序列化與合約表面的慣例定案。兩項變更最顯著。其一,MessagePack 合約序列化由位置式整數鍵改為 **property-name key**,使 JSON 與 MessagePack payload 由同一份以屬性名為準的慣例管轄,整數鍵佈局的長年 footgun 隨之消失。其二,API **合約介面依軸分入命名空間**,使介面層終於對齊實作層與訊息層早已採用的軸分法。兩項技術上皆屬破壞性,但目前無外部消費者,依 pre-stable 政策以 minor 發佈。序列化變更的設計理由見 [ADR-030](../adr/adr-030-messagepack-name-based-keys.md)。

## 破壞性變更

- **MessagePack 合約改用 property-name key(wire)** —— 框架長年以位置式整數 `[Key(n)]` 標註合約 DTO,序列化為緊湊的 MessagePack 陣列。此佈局有兩個長期隱患:集合 item 的建構子參數順序必須與 `[Key]` 順序完全一致(一旦漂移即在 wire 上靜默對調欄位),且 key 編號須跨繼承鏈手動協調。自 4.15.0 起,72 個合約型別改為 `[MessagePackObject(keyAsPropertyName: true)]` 並移除整數鍵:57 個 `Bee.Api.Core.Messages` request/response 型別(含 `ApiMessageBase.Parameters`,其 `[Key(0)]` 一併移除),與 15 個 `Bee.Definition` / `Bee.Api.Contracts` DTO 與非 `[Union]` 集合 item(`PackageUpdateInfo` / `PackageUpdateQuery`、`RecordFieldChange`、`CompanyInfo`、`DepartmentTree` / `DepartmentNode`、`PagingInfo` / `PagingOptions`、`AllowedCurrencyItem`、`CashRoundingItem`、`NumberFormatItem`、`CurrencyItem`、`UnitItem`、`SortField`、`Parameter`)。MessagePack payload 因而由位置式陣列改為屬性名 map,結構與 JSON payload 一致。此為破壞性 wire 變更;因 client 與 server 一同發佈且無外部消費者,今日無相容性成本,但 client 與 server 須跑相容版本 —— 新的 name-based formatter 無法讀取舊的位置式鍵 payload。此遷移為 **category-aware**,非全面轉換:`[Union]` 多型型別(`FilterNode` / `FilterCondition` / `FilterGroup`)維持整數鍵,因 `[Union]` 判別碼與 `keyAsPropertyName` 不相容;集合容器(`MessagePackCollectionBase` / `KeyCollectionBase` 子型別)走自訂 `CollectionBaseFormatter` / `ItemsForSerialization` proxy,不受鍵策略影響;`SerializableData*`(DataSet/DataTable wire plumbing)保持整數鍵。Phase 0 已證 `keyAsPropertyName` 在 reflection-only(AOT)路徑下亦正常 round-trip,故 MessagePack source generator 非前置條件。[ADR-030](../adr/adr-030-messagepack-name-based-keys.md)

- **合約介面依軸重組入命名空間(source)** —— `Bee.Api.Contracts` 套件長年將所有合約介面平放於根命名空間,即使對應的實作層(`Bee.Business.System` / `.Form` / `.AuditLog`)與訊息層(`Bee.Api.Core.Messages.System` / `.Form` / `.AuditLog`)早已軸分。這使介面層成為唯一還存在「合約實作已入軸、合約介面卻平放」不對稱的地方。4.15.0 將介面移入對應子資料夾 / 命名空間:System 軸(29 介面 + `PackageDelivery` / `PackageUpdateInfo` / `PackageUpdateQuery` DTO)入 `Bee.Api.Contracts.System`,Form 軸(12 介面:`IDelete` / `IGetData` / `IGetList` / `IGetLookup` / `IGetNewData` / `ISave` 各 Request/Response)入 `Bee.Api.Contracts.Form`,AuditLog 軸(13 介面 + `RecordFieldChange` DTO)入 `Bee.Api.Contracts.AuditLog`。根 `Bee.Api.Contracts` 命名空間現僅保留跨 BO、BO 無關的 `ExecFunc` 合約(`IExecFuncRequest` / `IExecFuncResponse`),鏡射訊息層根層的 `ExecFunc` 實作。此為純 source-level 變更:序列化實作類別的命名空間不變,對 wire 無影響。參照被搬移介面的外部消費者需更新 `using`(見下)。

## 變更

- 見上方兩則破壞性變更 —— 本版對使用者可見的表面完全是序列化與合約命名空間的收斂。

## 修正

- 無 user-facing 修正。(內部上,修好一個 sample factory 漏補 4.14.0 新增的 `CreateLogBusinessObject` override,並在 Blazor 測試專案修補一則新發布的 AngleSharp advisory —— 兩者皆不影響已發佈套件。)

## 升級指引

參照被搬移合約介面的外部消費者,將 `using` 更新至對應軸命名空間:

```diff
- using Bee.Api.Contracts;
+ using Bee.Api.Contracts.System;   // ILoginRequest, IPingRequest, …
+ using Bee.Api.Contracts.Form;     // IGetListRequest, ISaveRequest, …
+ using Bee.Api.Contracts.AuditLog; // 異動軸合約, RecordFieldChange
```

`ExecFunc` 合約(`IExecFuncRequest` / `IExecFuncResponse`)仍在根 `Bee.Api.Contracts` 命名空間,不需更改。

MessagePack wire 格式變更不需改程式碼,但 client 與 server 必須跑相同(或相容)版本 —— 以舊的位置式整數鍵寫入的 payload 無法被新的 name-based formatter 讀取。
