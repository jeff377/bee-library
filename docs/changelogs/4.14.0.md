# Bee.NET 4.14.0 — Detailed Release Notes

[English](4.14.0.md) ・ [繁體中文](4.14.0.zh-TW.md) ・ [← CHANGELOG](../../CHANGELOG.md)

This file is the detailed companion to the `[4.14.0]` entry in [CHANGELOG.md](../../CHANGELOG.md). The top-level changelog lists the headline items in one line each; this file keeps the full narrative — the *why*, the constraints, and the design context behind each change.

> Bee.NET remains in pre-stable evolution. This release lands two subsystems and one cross-cutting normalization. The **expression & rule engine** moves field computation and validation out of hand-written business objects and into declarative `FormSchema` definitions — a computed field is a `ValueExpression`, a validation is a `FormRule`, and the same portable engine runs on the server (authoritative, before save) and in the Avalonia client (live preview). The **audit-trail subsystem** records six independent log axes into `st_log_*` tables and exposes a read side for querying them. Finally, in-memory `DataSet` column names are canonicalized to **lowercase** so a field name is spelled the same way at every layer — definition, database, DataSet, expression, UI — eliminating the case-sensitive name-matching problems that surfaced while wiring up the expression engine. That last change is wire-visible and therefore breaking; the pre-stable policy ships it as a minor. Design rationale lives in [ADR-028](../adr/adr-028-expression-rule-engine.md), [ADR-027](../adr/adr-027-audit-trail.md), and [ADR-029](../adr/adr-029-lowercase-field-names.md).

## Breaking changes

- **`DataSet` column names are lowercase (wire & public API)** — the framework historically stored in-memory `DataSet` column names in UPPERCASE, a normalization introduced years ago to satisfy a case-sensitive UI-binding path. That casing leaked onto the wire, so JSON / MessagePack payloads carried uppercase keys such as `SYS_ROWID`. As of 4.14.0 the canonical spelling is lowercase at every layer: `DataTableExtensions.AddColumn` lowercases (invariant, fixing a latent Turkish-I hazard), and the read boundary in `DbAccess` calls `LowercaseColumnNames` (renamed from `UppercaseColumnNames`) to fold each provider's native casing — Oracle returns uppercase, PostgreSQL lowercase — to a single lowercase form. The JSON / MessagePack converters emit the `DataColumn.ColumnName` directly, so payload keys are now lowercase (`sys_rowid`). **.NET consumers are unaffected**: `DataColumnCollection` lookups, `GetFieldValue`, and the expression engine all match names case-insensitively (the engine binds by `FormField.FieldName`). Only two categories need action — external JS/TS clients that read a `DataSet` by literal uppercase key, and any code that called the renamed `UppercaseColumnNames`. See [ADR-029](../adr/adr-029-lowercase-field-names.md) and the migration plan for the full rationale and the multi-DB verification.

## Added

- **Expression & rule engine (`Bee.Expressions`, new package)** — a portable, sandboxed evaluator (`IExpressionEvaluator` / `DynamicExpressoEvaluator`, backed by DynamicExpresso, MIT). Expressions are a C# subset (`unit_price * quantity`, `amount > 0`, `status == "Approved"`); the sandbox exposes only field variables, a curated helper set (`Today` / `Now` / `IsNullOrEmpty` / `Guid`) and primitive types — any other identifier fails at parse time. Each expression compiles once and is cached by text + parameter signature. `ExpressionPolicy` centralizes the type/null policy (`FieldDbType` → CLR, `DBNull` → the type's default so `unit_price * qty` treats a blank operand as zero) so the client preview and the server save produce identical results. The engine also exposes referenced-variable analysis to drive the "which edit forces which recompute" dependency graph. [ADR-028](../adr/adr-028-expression-rule-engine.md)

- **Field expressions & rules on the schema (`Bee.Definition`)** — `FormField` gains `ValueExpression` (a computed field, recomputed and rounded before save) and `DefaultValueExpression` (a new-row default produced by an expression). `FormSchema` gains a `FormRule` / `FormRuleCollection`: each rule has an optional `When` applicability guard, a `Condition` that must hold, a `Message` (literal or language-resource key), a `Trigger` (`BeforeSave` | `BeforeDelete`), an optional `TargetTable`, `Enabled`, and `Order`. The shared `FormExpressionCalculator` (in `Bee.Definition.Forms`, mirroring `FormRowDefaults`) holds the row-level compute/validate logic so the server and the UI client run one code path and cannot drift.

- **BO lifecycle template methods (`Bee.Business`)** — `FormBusinessObject.Save` / `Delete` are refactored into template methods: authorization, write-scope, and change-audit stay fixed in the orchestration layer, while `DoBeforeSave` / `DoSave` / `DoAfterSave` (and the delete equivalents) are `protected virtual` extension points. The base `DoBeforeSave` applies, via `IFormRuleProcessor`, the schema-driven default-value expressions, then the computed fields (full precision, then rounded through `NumberFormatResolver.RoundByKind`), then the `BeforeSave` validation rules. A general CRUD form needs **zero BO code**; a custom BO overrides a `Do*` method and calls `base` first. The Northwind `OrderBO` is migrated as the worked example: line amount becomes a `ValueExpression`, the required-field checks become `FormRule`s, and only the genuinely imperative parts (order-number sequence, stored-status guard) stay in code.

- **Client-side live preview (`Bee.UI.Avalonia`)** — `FormLiveComputation` recomputes computed fields as the user edits, subscribed to the unified `FormDataObject.FieldValueChanged`. Master editors refresh through their own binding; a detail grid's realized cells are refreshed via a dispatcher-posted `RefreshRows`. A dependency graph gates recomputation, a re-entrancy guard prevents the write-back from re-triggering, and any evaluation failure degrades the preview to a no-op (the server remains authoritative). Tier 2 rounding aligns the preview to the server's decimals: `FormView` resolves a `RoundingContext` from `ClientInfo.Company` plus the currency/unit masters and injects it into the calculator and the amount/quantity editors. AOT was verified on the reflection-only path (`IsDynamicCodeSupported=false`), so the engine works on WASM / mobile without disabling live preview.

- **Audit-trail subsystem (`Bee.Business` / `Bee.Repository`)** — six independent log axes write to `st_log_*` tables (`login`, `change`, `access`, `anomaly_api`, `anomaly_db`) in the `log` database category. `IAuditLogWriter` batches writes on a background channel so instrumentation never blocks the request. The change axis captures a `DataSet` DiffGram (both current and original values) on save/delete, giving a full before/after image. See [ADR-027](../adr/adr-027-audit-trail.md) for the six-axis design and the `st_log_*` naming.

- **Audit log query read side (`Bee.Business` / `Bee.Api.*`)** — `GetChangeLog` / `GetChangeDetail` drive a two-stage change browser (a per-record list, then the field-level DiffGram detail of one entry); parallel list queries cover the login, access, and anomaly axes; and an anomaly summary returns a `Summary` plus Top-N aggregation. The redundant `GetRecordHistory` introduced mid-cycle was removed once `GetChangeLog` + `GetChangeDetail` covered it, so it never shipped.

- **Front-end permission capability (`Bee.UI.Avalonia` / `Bee.UI.Core`)** — the `EnterCompany` capability snapshot now degrades UI elements client-side: a field with no Read is hidden, a field with no Update renders read-only (this is UX degradation only; the server stays the authoritative boundary). `ClientInfo` caches the entered `CompanyInfo` (`ClientInfo.Company`), and `ClientDefineAccess` gains `GetCurrencySettingsAsync` / `GetUnitSettingsAsync`. Record-scope permission also gains support for multiple Owner / Dept fields, evaluated as an OR union.

## Changed

- **SQL Server `DateTime` → `datetime2(7)` (`Bee.Db`)** — SQL Server `DateTime` columns now emit `datetime2(7)` instead of `datetime`, gaining sub-millisecond precision and the pre-1753 date range. The `datetime2` parameter rewrite is scoped to SQL Server only, so PostgreSQL / Oracle seeding is unaffected (a mid-cycle regression there was fixed).

- **String-key comparisons → `OrdinalIgnoreCase` (`Bee.Base`)** — case-insensitive comparisons over string keys (collection keys, field names, ids) converged on `OrdinalIgnoreCase` instead of culture-aware comparisons, making them culture-independent and fastest, and avoiding the Turkish-I hazard where `CurrentCultureIgnoreCase` treats `ID` ≠ `id` under a Turkish locale.

## Fixed

- **Expression variable binding & value coercion (`Bee.Expressions`)** — three related fixes surfaced while dogfooding the engine on the Northwind order form:
  - The variable map is now keyed by the declared `FormField.FieldName`, not the raw `DataColumn.ColumnName`. Because `AddColumn` stored column names uppercase and DynamicExpresso identifiers are case-sensitive, an expression referencing the declared lowercase `quantity` failed to resolve against a `QUANTITY` column (an unknown-identifier error that surfaced as a `-32000` internal error on save). Keying by the declared field name decouples expression identifiers from the DataSet's storage casing. (The 4.14.0 lowercase normalization removes the root mismatch as well.)
  - `ExpressionPolicy.CoerceValue` now handles string-typed `Guid` and `byte[]` columns — a wire-deserialized or SQLite-sourced cell frequently arrives as a string (SQLite stores GUIDs as TEXT). `Convert.ChangeType` throws for these non-`IConvertible` types; the coercion parses them explicitly.
  - An empty-string GUID (an unset key, a text-typed GUID column with no selection) maps to `Guid.Empty` rather than throwing from `Guid.Parse("")`, matching the null/DBNull default policy.
