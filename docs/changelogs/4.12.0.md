# Bee.NET 4.12.0 — Detailed Release Notes

[English](4.12.0.md) ・ [繁體中文](4.12.0.zh-TW.md) ・ [← CHANGELOG](../../CHANGELOG.md)

This file is the detailed companion to the `[4.12.0]` entry in [CHANGELOG.md](../../CHANGELOG.md). The top-level changelog lists the headline items in one line each; this file keeps the full narrative — the *why*, the constraints, and the design context behind each change.

> Bee.NET remains in pre-stable evolution. The theme of this release is "the Avalonia control family goes mobile": `Bee.UI.Avalonia` becomes responsive for phone / narrow viewports, and `Bee.Definition` types are made compatible with the AOT reflection-only XmlSerializer. Together these are what let the same shared Avalonia UI run unchanged as **iOS** and **Android** single-view heads (validated end-to-end on the Northwind demo). There are **no breaking changes** — call syntax and on-disk XML format are unchanged.

### Added

- **`FormView` responsive layout (`Bee.UI.Avalonia`)** — on a narrow viewport the whole form adapts: the master field area reflows from `FormLayout.ColumnCount` columns to a single column, and detail `GridControl`s switch from inline `InCell` editing to the read-only-grid + `EditForm` flow. A single `_isCompact` state drives both; the form is only rebuilt when the viewport crosses `CompactWidthThreshold` (a new styled property, default 600 DIP), so width changes within a band do not thrash. Width is observed through `OnPropertyChanged` on the inherited `Bounds` direct property (class handlers do not fire for it). `DetailEditMode` is retained as the wide-screen preference (default `InCell`), so desktop behaviour is unchanged. No head wiring is required — any head renders compact automatically when narrow.
- **`ListView` card layout on narrow viewports (`Bee.UI.Avalonia`)** — the wide column grid clips fields and forces horizontal scrolling on a phone. Below `CompactWidthThreshold` the grid is replaced by a `ListBox` of one card per row, each visible column rendered as a `caption  value` line read straight off the `DataRowView` (no reflection binding, so it is safe under iOS AOT). Selection still drives the existing toolbar-enable flow. Wide desktop / browser viewports are unaffected.
- **`RowEditPanel` (EditForm) responsive + resizable `RowEditDialog` (`Bee.UI.Avalonia`)** — the row-edit panel reflows 1 ↔ 2 columns by the host top-level width (screen width, not the dialog's own width, so a small `SizeToContent` desktop window is not misread as compact). The desktop native `RowEditDialog` window is now `CanResize=true`, switching `SizeToContent` back to `Manual` after open so the initial content-sized window can still be dragged to resize.

### Fixed

- **Definition types compatible with the AOT reflection-only XmlSerializer (`Bee.Definition`)** — on AOT targets (iOS, and any NativeAOT/MAUI-iOS target) there is no `Reflection.Emit`, so `XmlSerializer` falls back to its reflection-only reader, which crashed when loading any Bee XML definition. Two incompatibilities were removed: (1) collections exposed more than one public instance `Add`, so the reader's `Type.GetMethod("Add")` threw `AmbiguousMatchException` — each definition collection now keeps a single public instance `Add(T)` (base interface `Add(I…CollectionItem)` is an explicit interface implementation; the per-collection convenience `Add(...)` overloads are now extension methods in the same namespace, so caller `.Add(...)` syntax is unchanged); (2) owner-coupled collections lacked a public parameterless constructor that `Activator.CreateInstance` needs, so 12 of them gained `public Xxx() : base()`. The desktop / browser (WASM) code-gen path was always fine and is unaffected; the XML format is byte-for-byte unchanged (verified by the Definition + Base round-trip test suites). See [ADR-025](../adr/adr-025-define-types-aot-xmlserializer-compat.md).
- **`RowEditDialog` overlay on single-view hosts (`Bee.UI.Avalonia`)** — row editing previously special-cased the browser to use an overlay and opened a native `Window` everywhere else. iOS / Android single-view hosts do not support a native `Window`, so editing a detail row crashed. The decision is now capability-driven: a native modal opens only when the top-level is a `Window` (desktop classic-window lifetime); otherwise (browser / iOS / Android single-view) it renders through the `OverlayLayer`. The crash-prone free-standing-window fallback was removed.
- **`FormView` body scrolls vertically (`Bee.UI.Avalonia`)** — the content was a non-scrolling vertical stack, so a tall single-column (compact) layout overflowed the viewport and the lower controls were unreachable. The error label and toolbar are now pinned at the top (Save / Cancel / Back always reachable) via a `DockPanel`, with the form body inside a vertical `ScrollViewer`. Detail-grid behaviour is unchanged.
- **`GridControl` lookup cell magnifier icon (`Bee.UI.Avalonia`)** — an editable detail-grid lookup cell (which bypasses the DataGrid edit pipeline per ADR-021 and opens its dialog from the display template) rendered plain text with no affordance, inconsistent with the master-form `ButtonEdit`. The editable branch now shows a muted open-dialog magnifier icon; read-only / list modes and `ButtonEdit` are unchanged.
