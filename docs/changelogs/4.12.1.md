# Bee.NET 4.12.1 — Detailed Release Notes

[English](4.12.1.md) ・ [繁體中文](4.12.1.zh-TW.md) ・ [← CHANGELOG](../../CHANGELOG.md)

This file is the detailed companion to the `[4.12.1]` entry in [CHANGELOG.md](../../CHANGELOG.md). The top-level changelog lists the headline items in one line each; this file keeps the full narrative — the *why*, the constraints, and the design context behind each change.

> Bee.NET remains in pre-stable evolution. 4.12.0 made the `Bee.Definition` types *deserialize* under the AOT reflection-only XmlSerializer (no `Reflection.Emit`); this patch closes the other half of the same problem — *trimming*. On a trimmed iOS / Android Release build the linker removes the "apparently unused" definition metadata, so this release ships an embedded trimmer descriptor that preserves it. There are **no breaking changes** — call syntax and on-disk XML format are unchanged.

### Fixed

- **Embedded trimmer descriptor preserves the definition graph under trim / AOT (`Bee.Definition`)** — the Avalonia iOS / Android heads are remote JSON-RPC clients: a `FormSchema` (and the other definition types) arrives as an XML string carried inside the JSON-RPC payload and is reconstructed on the device through `XmlCodec.Deserialize<T>`. On a trimmed Release build the linker sees no static caller for the parameterless constructors, property setters, and strongly-typed collection `Add(T)` members — the only caller is the reflection path, which the trimmer cannot follow — and strips that metadata. At runtime this surfaces as the misleading `XmlSerializeErrorDetails, 2, 2` (line 2, column 2 is the root element opening; the XML is fine, the type metadata was trimmed).

  This release adds `src/Bee.Definition/ILLink.Descriptors.xml`, embedded as the logical resource `ILLink.Descriptors.xml` — the convention the .NET trimmer scans for — which wildcard-roots `Bee.Definition.*` and `Bee.Base.Collections.*` with `preserve="all"`. Because it is embedded in the shipped assembly, the trimmer applies it automatically for **every** downstream trimmed / AOT consumer: external apps building trimmed mobile heads inherit the fix without authoring their own linker file, and no `PackageReference` beyond the normal transitive dependency on `Bee.Definition` is required.

  Measured on an Android emulator full-trim (`TrimMode=full`) build: without the descriptor the linker removes ~57% of `Bee.Definition` (212 KB → 91 KB), and a definition shape whose serialization members are not otherwise referenced can lose them; with the descriptor the assembly is preserved (~208 KB) and the `FormSchema` round-trip is intact. Validated end-to-end without a physical device: Android emulator full trim (round-trip passes), an iOS device-target Release AOT build (compiles clean, descriptor honoured by the iOS linker — linked `Bee.Definition.dll` stays at full size), and the reflection-only serializer path forced on the iOS simulator (`RuntimeFeature.IsDynamicCodeSupported=false`, the same path device AOT is locked onto) where the round-trip passes.

  The broad wildcard root (rather than per-type `[DynamicallyAccessedMembers]`) is deliberate: the definition graph is polymorphic and subtype-dense, so a per-type annotation pass is error-prone — a wildcard covers the whole family at once. Related: [ADR-025](../adr/adr-025-define-types-aot-xmlserializer-compat.md) (the sibling reflection-only-deserialization fix shipped in 4.12.0). Full investigation and the half-A / half-B framing are in [plan-mobile-release-trim-safe.md](../plans/plan-mobile-release-trim-safe.md).
