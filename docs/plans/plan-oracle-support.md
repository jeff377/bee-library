## 背景

`DatabaseType.Oracle` 自 enum 建立以來就是預埋值，但目前無對應 dialect / provider 實作。本計畫補完到與 SQLite 等同的支援等級，但 Oracle 方言怪癖較多（IDENTITY/SEQUENCE、`NUMBER` 統一型別、`VARCHAR2` 字節 vs 字符語意、reserved word 較多），需逐項決策後實作。

策略原則（CI 純語法、整合本機）見總綱。

## 目標

1. `DatabaseType.Oracle` 透過 `DbProviderRegistry.Get()` 與 `DbDialectRegistry.Get()` 取得有效實作
2. CREATE TABLE / ALTER TABLE / Rebuild / Form 四類 SQL 產生器在純語法測試中通過
3. 本機設好 `BEE_TEST_CONNSTR_ORACLE` 後，所有 `[DbFact(DatabaseType.Oracle)]` 整合測試通過
4. CI 不變動（無 service container），未設環境變數時整合測試自動 skip

## 不涵蓋

- **Oracle 12c 以下**：本計畫鎖定 19c+，理由見下節
- **Real Application Clusters / Data Guard / RAC 拓撲**：純單庫支援
- **`PL/SQL` 儲存程序**：框架層走 ADO.NET 直連 SQL，不涉及 PL/SQL 包裝
- **特殊 tablespace / partitioning 配置**：CREATE TABLE 不指定 tablespace（用使用者預設）
- **NLS 國際化深度設定**：採用合理預設（`NLS_LENGTH_SEMANTICS=CHAR`），不引入運行期切換
- **效能調優**：bind variable 快取、bulk collect 等留給後續

## 套件選擇

採用 **`Oracle.ManagedDataAccess.Core`**（NuGet：`Oracle.ManagedDataAccess.Core`）。

| 比較 | Oracle.ManagedDataAccess.Core | Oracle.DataAccess.Client |
|---|---|---|
| 形式 | 純 .NET | 需安裝 Oracle Client（原生 binary） |
| 跨平台 | ✅ Windows / Linux / macOS | ❌ Windows only |
| 授權 | Oracle Free Use Terms | 同左 |
| 版本支援 | 11g+ | 同左 |

選擇理由：純 .NET 無原生依賴、跨平台（與本專案 .NET 10 / Linux CI 一致）、官方維護。雖授權非 MIT 但對 NuGet 散佈無限制（Oracle Free Use Terms 允許再散佈）。

加入位置：[`tests/Bee.Tests.Shared/Bee.Tests.Shared.csproj`](../../tests/Bee.Tests.Shared/Bee.Tests.Shared.csproj)。`Bee.Db` 本身仍保持 ADO.NET 無關。

## 版本假設：Oracle 19c+

理由：
- **`GENERATED BY DEFAULT AS IDENTITY`**：12c 才支援；舊版需 SEQUENCE + trigger 模擬，會把方言層搞髒
- **128 字元識別符**：12c R2 才放寬（舊版 30 字元限制易踩雷）
- **`OFFSET m ROWS FETCH NEXT n ROWS ONLY`**：12c+ 標準分頁；舊版需 `ROWNUM` 包巢狀查詢
- **19c 是 LTS（Long Term Support）**：21c 為 Innovation Release，23ai 是當前 LTS（含原生 BOOLEAN）；19c 仍最普及，Premier Support 至 2027
- **EOL 視窗**：12c R2 已 EOL（2022-03），11g 更早 EOL；新框架無相容必要

文件中明確標註此前提。如目標環境為 23ai，本計畫產出仍可用，但不主動利用 23ai 新特性（如原生 BOOLEAN）以保留向下相容。

## 方言決策

| 項目 | 決策 | 備註 |
|---|---|---|
| Parameter prefix | `:` | 已預埋於 [`DbFunc.cs:22`](../../src/Bee.Db/DbFunc.cs) |
| Identifier quoting | `"name"` | 已預埋於 [`DbFunc.cs:57`](../../src/Bee.Db/DbFunc.cs)；Oracle reserved word 多，identifier 一律 quote 為安全 |
| 識別符大小寫 | quoted 識別符**大小寫敏感** | 框架統一 quote 後等於要求大小寫一致；FormSchema 表/欄名規範交由上層約定（推薦全小寫） |
| 識別符長度 | 上限 128（19c+） | 與 SQL Server / PostgreSQL 對齊，不額外截斷 |
| 字符集 | DB 預設（通常 `AL32UTF8`） | CREATE TABLE 不指定字符集 |
| `NLS_LENGTH_SEMANTICS` | 一律 `CHAR` 語意 | `VARCHAR2(100 CHAR)` 而非 `VARCHAR2(100)` 或 `(100 BYTE)`；避免中文截斷 |
| Session NLS（case-insensitive） | `NLS_COMP=LINGUISTIC` + `NLS_SORT=BINARY_CI` | `OracleProvider` 開連線後立即 `ALTER SESSION` 套用，**ERP 字串查詢預設 CI 比對**（如查 `jeff` 命中 `Jeff`）。詳見下節「連線初始化」與總綱「Case-insensitivity 策略」 |
| AutoIncrement | `GENERATED BY DEFAULT AS IDENTITY` | 12c+ 標準語法；不採 SEQUENCE + trigger |
| Boolean | `NUMBER(1) DEFAULT 0` + CHECK | 19c 無原生 `BOOLEAN`（23ai 才有）；統一用 NUMBER(1) |
| 整數 | `NUMBER(10)` (int) / `NUMBER(19)` (bigint) | Oracle 無原生 INT/BIGINT，全為 `NUMBER` |
| 小數 | `NUMBER(p, s)` | 對應 `decimal(p, s)` |
| 字串（短） | `VARCHAR2(n CHAR)` | 上限 4000 byte（標準）或 32767（`MAX_STRING_SIZE=EXTENDED`，需 DBA 設定） |
| 字串（長） | `CLOB` | 超過 4000 byte 一律 CLOB；不依賴 EXTENDED 模式 |
| 二進位（短） | `RAW(n)` | 上限 2000 byte |
| 二進位（長） | `BLOB` | 同上策略 |
| UUID | `RAW(16)` | 比 `CHAR(36)` 省空間且對齊 `SYS_GUID()` 回傳型別；seed 表達式即 `SYS_GUID()` |
| DateTime | `TIMESTAMP(6)` | 不採 `DATE`（精度只到秒、語意混亂） |
| Default expressions | `SYS_GUID()`, `SYSTIMESTAMP` | seed 對應；`SYSTIMESTAMP` 含時區，比 `SYSDATE`/`CURRENT_TIMESTAMP` 更明確 |
| 分頁 | `OFFSET m ROWS FETCH NEXT n ROWS ONLY` | 12c+ 標準語法；不採 `ROWNUM` 巢狀 |
| Schema 概念 | Schema = User | 連線後預設 schema 為登入 user；跨 schema 用 `"USER"."TABLE"` |
| Schema 查詢 | `ALL_TAB_COLUMNS` / `ALL_INDEXES` / `ALL_CONSTRAINTS` | 採 `ALL_*`（含當前 user 可見的所有 schema）而非 `USER_*`（僅當前 user）；多 schema 場景較通用 |
| ALTER 能力 | 多數操作支援，但比 SQL Server 嚴格 | `MODIFY COLUMN` 對型別變更有限制（如 NUMBER 縮減 precision）；以 SQL Server 為基準微調 |
| 分號結尾 | **不加** | Oracle SQL\*Plus 才需要 `;`；ADO.NET 透過 `OracleCommand` 執行單一 statement 不需要結尾分號（PL/SQL 區塊例外，本層不涉及） |

### 連線初始化

`OracleProvider` 在每次開連線後（`DbConnection.Open()` 之後、回傳給 `DbAccess` 之前）執行：

```sql
ALTER SESSION SET NLS_COMP='LINGUISTIC' NLS_SORT='BINARY_CI'
```

效果：當前 session 內所有字串比對（`=`、`LIKE`、`ORDER BY`）皆為 case-insensitive。實作位置：Oracle Provider 註冊的連線工廠 lambda 內（詳見「Registry 註冊」段）。

不採用 column-level collation 的理由：Oracle 12.2+ 才支援 column-level collation 且需 DB 層級 `MAX_STRING_SIZE=EXTENDED`，不夠通用；session-level NLS 是 Oracle 標準做法、無前置設定要求、不污染 CREATE TABLE 語法。

未來若需要其他 session 級設定（如 `NLS_DATE_FORMAT`、`NLS_NUMERIC_CHARACTERS`），統一在此初始化點集中。

### Reserved Word 風險

Oracle 的 reserved word 與 SQL Server / PostgreSQL 重疊度高但有額外項目（如 `COMMENT`、`SIZE`、`LEVEL`、`UID`、`SESSION` 等）。框架層面策略：**所有識別符一律 quote**——`DbFunc.QuoteIdentifier` 已對 Oracle 套用 `""`，運作上等同強制大小寫敏感。FormSchema 命名規範（全小寫）需明確約定，否則 `"Order"` 與 `ORDER`（reserved word）會產生意料外行為。

> 風險點：如果有 user 在 schema 設計時使用混合大小寫名稱（如 `"OrderItem"`），其他 DB 因不大小寫敏感能 work，Oracle 卻必須精確匹配 quote 後的字串。建議在驗收測試中明確覆蓋此情境。

## 測試規劃

### CI（純語法）

- 執行範圍：`Oracle*Tests` 系列（4 個檔）
- 依賴：無連線、無 driver 載入
- 預期：與既有 SQLite / SQL Server 純語法測試同樣 ms 級執行時間

### 本機（整合）

- **本機環境**：Oracle Database Free 23ai 透過 Docker
  ```bash
  docker run -d --name oracle23ai \
    -p 1521:1521 \
    -e ORACLE_PWD=BeeTest_Pass123 \
    container-registry.oracle.com/database/free:latest
  # 啟動需 1-2 分鐘；ready 後可建立 testuser
  ```
- **Docker Desktop 資源配置建議**：
  - RAM：上調至 **12–16GB**（Oracle 23ai container 預估占用 2.5–3.5GB；同時跑 SQL Server + PostgreSQL + MySQL + Oracle 四 container 共需 ~6GB）
  - 設定路徑：Docker Desktop → Settings → Resources → Memory
  - 32GB 主機調至 16GB 仍有 16GB 留給 macOS 本體與 IDE
- 磁碟：image 占用 5–7GB
- **container 啟動時間長**（1–2 分鐘）的應對：
  - `test.sh` **不**加自動偵測（避免每次 test 都等 container）
  - 開發者保持 container 常駐，僅在重啟系統時手動拉起
  - 整合測試在 dialect 大改時跑，平時靠純語法測試把關
- 啟用方式：`.runsettings` 取消註解 `BEE_TEST_CONNSTR_ORACLE` 並填入連線字串

### 備援：Oracle Cloud Always Free

若本機 Docker 暫時不可用（如出差、Mac 維修期間），可改用 Oracle Cloud Always Free 的 Autonomous Database：
- 永久免費（2 instance、20GB 儲存）
- 連線需下載 wallet 配置 connection string
- 閒置 7 天會自動 stop（手動 start 即恢復、資料保留）
- 透過修改 `BEE_TEST_CONNSTR_ORACLE` 即可切換，方言層程式碼不需變動

### 不涵蓋的測試類型

- **23ai 原生 BOOLEAN 行為**：本計畫一律走 NUMBER(1)，不分支
- **跨 schema 查詢**：framework 假設單 schema；多 schema 場景留給應用層自行處理 `"OWNER"."TABLE"` 限定詞

## 風險與緩解

| 風險 | 影響 | 緩解 |
|---|---|---|
| Oracle container 啟動慢、本機開發體感差 | 中 | 文件標註「Oracle 整合測試在 dialect 大改時跑」、平時靠純語法測試 |
| `OracleTableSchemaProvider` 的 `ALL_TAB_COLUMNS` 查詢結果欄位名 / 型別字串解析較繁雜 | 中 | 參考 `PgTableSchemaProvider`（同走 INFORMATION_SCHEMA-like）+ Oracle 文件 |
| Boolean → NUMBER(1) 的 round-trip：寫入 `true` / `false` 是否能正確映射回 .NET `bool` | 中 | 在 type mapping 與 `DbAccess` 讀取邏輯加測試覆蓋；MySQL 也是同樣 pattern（TINYINT(1)），可借鏡 |
| Reserved word 衝突（如 schema 命名混到 `COMMENT`、`SIZE`） | 中 | identifier 一律 quote；驗收測試包含 reserved word 表/欄名情境 |
| 識別符大小寫敏感（quoted 後）造成跨 DB 行為不一致 | 中 | 文件約定 FormSchema 命名全小寫；測試覆蓋大小寫一致性 |
| `DbProviderRegistry` 可能無 connection-open hook，需小重構支援 Oracle 連線初始化（NLS 設定） | 中 | 動工初期先驗證註冊 pattern；若需新增 hook，影響面只限新增介面方法、現有四 DB 不需修改 |
| `VARCHAR2` 4000 byte 上限導致長字串需 CLOB，且 CLOB / VARCHAR2 在 ADO.NET 取值方式不同 | 中 | 在 `OracleTypeMapping` 自動依長度分流（>=4000 一律 CLOB）；DbAccess 讀取 CLOB 已有標準 pattern |
| `DATE` vs `TIMESTAMP` 混用造成精度遺失 | 低 | 統一用 `TIMESTAMP(6)`，禁用 `DATE` |
| `Oracle.ManagedDataAccess.Core` 在 macOS arm64 / Linux arm64 的支援度 | 低 | 套件已支援 .NET 6+ 跨平台；本機 macOS Apple Silicon 需驗證一次 |
| Oracle 連線授權（`testuser` 需要 CREATE TABLE / DROP TABLE 等權限）| 低 | 文件提供建立 testuser 的 SQL 腳本（`GRANT CONNECT, RESOURCE`）|
