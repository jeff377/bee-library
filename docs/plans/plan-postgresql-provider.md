# 計畫：Bee.Db 加入 PostgreSQL Provider（第二 RDBMS 支援）

**狀態：📝 擬定中（已依 ALTER-based 升級計畫落地後的架構重整，待最終確認後動工）**

> **先決條件已解**：[`plan-alter-based-upgrade.md`](plan-alter-based-upgrade.md) 已於 2026-04-25 完成（v4.1.0）。該計畫引入的 `TableUpgradeOrchestrator` / `ITableAlterCommandBuilder` / `SqlTableRebuildCommandBuilder` / `SqlSchemaHelper` / `SqlAlterCompatibilityRules` 都是新的 provider-specific 耦合點，本計畫的抽象層萃取需一併覆蓋這些新面向（見下方「主要耦合點」與「技術設計 1.2」）。

## 背景

Bee.NET 目前的 `Bee.Db` 僅支援 SQL Server。為擴大在 ERP 市場的適用面（Odoo、ERPNext、iDempiere 等開源 ERP 生態），並驗證抽象層是否足夠紮實，本計畫規劃加入 **PostgreSQL** 作為第二個 RDBMS Provider。

選擇 PostgreSQL 而非 MySQL/MariaDB 的理由：

- PostgreSQL 與 SQL Server 差異夠大（識別字大小寫、`information_schema` vs `sys.*`、型別系統、Sequence vs IDENTITY），能**逼出現有抽象層的技術債**，避免等到第三個 provider 才爆
- MySQL 行為與 SQL Server 接近（quoting、大小寫），做為第二個反而會**掩蓋抽象破綻**
- Npgsql driver 成熟，支援 `@name` 參數命名，不需改 `DbCommandSpec` 現有 API
- Docker image 輕量（< 150 MB，啟動 < 3 秒），適合現有 `./test.sh` 模式

| `DbCommandSpec` 佔位符 | `src/Bee.Db/DbCommandSpec.cs` | `{0}` / `{Name}` 佔位符 provider-agnostic，依 `DatabaseType` 轉換前綴 |

| `TableUpgradeOrchestrator` | `src/Bee.Db/Schema/TableUpgradeOrchestrator.cs` | Provider-agnostic 聚合層；但內部仍直接 `new` SqlServer builder（見下方耦合點） |
| `TableSchemaDiff` / `TableChange` / `UpgradePlan` | `src/Bee.Db/Schema/` | 結構化 diff 與執行計畫，完全 provider-agnostic |
| `FieldDbType` 抽象型別 | `src/Bee.Base/Data/FieldDbType.cs` | 14 種中性型別，provider 負責雙向映射 |

### 主要耦合點（須改造）

| 耦合點 | 路徑 | 問題 |
|--------|------|------|
| `TableSchemaBuilder` 直接 `new SqlTableSchemaProvider` | `src/Bee.Db/Schema/TableSchemaBuilder.cs` (line ~43) | 未透過工廠路由，無法依 `DatabaseType` 切換 |
| `TableUpgradeOrchestrator` 預設建構子直接 `new SqlTableAlterCommandBuilder` | `src/Bee.Db/Schema/TableUpgradeOrchestrator.cs` (line ~21) | 雖已有 DI constructor overload，預設路徑仍硬寫 SQL Server |
| `TableUpgradeOrchestrator.BuildCreatePlan` 直接 `new SqlCreateTableCommandBuilder` | 同上 (line ~110) | 新表 CREATE 語法硬寫 SQL Server |
| `TableUpgradeOrchestrator.BuildRebuildPlan` 直接靜態呼叫 `SqlTableRebuildCommandBuilder.GetCommandText` | 同上 (line ~118) | rebuild 路徑硬寫 SQL Server |
| `SqlTableRebuildCommandBuilder` 內部再 `new SqlCreateTableCommandBuilder` | `src/Bee.Db/Providers/SqlServer/SqlTableRebuildCommandBuilder.cs` (line ~45) | Provider 內聚相依 OK；但代表 PG 需自備對應的 rebuild builder |
| `SqlFormCommandBuilder` 硬轉 `DatabaseType.SQLServer` | `src/Bee.Db/Providers/SqlServer/SqlFormCommandBuilder.cs:49` | 該 Builder 本應只處理 SQL Server；問題是上層取用時沒有路由機制 |
| `DbFunc.GetSqlDefaultValue` 回傳 `getdate()` / `newid()` | `DbFunc.cs` | SQL Server 特有預設值寫死；PostgreSQL 對應為 `CURRENT_TIMESTAMP`、`gen_random_uuid()` |
| `SqlTableSchemaProvider` 查 `sys.tables` / `sys.columns` | `src/Bee.Db/Providers/SqlServer/SqlTableSchemaProvider.cs` | 與 `fn_listextendedproperty` 同屬 SQL Server 特有 catalog view |
| `SqlCreateTableCommandBuilder` DDL 語法 | `src/Bee.Db/Providers/SqlServer/SqlCreateTableCommandBuilder.cs` | `IDENTITY`、`[nvarchar](n)`、`sp_addextendedproperty` 等全屬 SQL Server 方言（ALTER 計畫後此類只負責 CREATE） |
| `SqlAlterCompatibilityRules` 型別相容性表 | `src/Bee.Db/Providers/SqlServer/SqlAlterCompatibilityRules.cs` | 跨 family / AutoIncrement / narrowing 規則為 SQL Server 語意；PG 需自備（型別 family 在 PG 可能略有差異，如無 nvarchar） |
| `SqlSchemaHelper` 識別字 quoting 與型別映射 | `src/Bee.Db/Providers/SqlServer/SqlSchemaHelper.cs` | `[nvarchar](n)` / `[bit]` / `IDENTITY(1,1)` 等硬寫；PG 需對應 `"col"` / `VARCHAR(n)` / `BOOLEAN` / `GENERATED BY DEFAULT AS IDENTITY` |
| `SqlExtendedPropertyCommandBuilder` | `src/Bee.Db/Providers/SqlServer/SqlExtendedPropertyCommandBuilder.cs` | `sp_addextendedproperty` 為 SQL Server 特有；PG 以 `COMMENT ON ...` 對應 |
| 尚無 `ITableSchemaProvider` 介面 | — | `SqlTableSchemaProvider` 是具體類別，未萃取抽象 |
| 尚無 `ITableRebuildCommandBuilder` 介面 | — | `SqlTableRebuildCommandBuilder` 為 `internal static`，ALTER 計畫留下的 provider-specific 類別 |
| 尚無 `IDialect` / Provider 路由器 | — | 沒有「依 `DatabaseType` 取得對應 builder」的集中點 |

## 目標與範圍

### 目標

1. `Bee.Db` 能透過 `DatabaseType.PostgreSQL` 建立連線、執行查詢、CRUD、DDL、Schema 比對升級
2. SQL Server 行為**完全不變**（零回歸）
3. 測試基礎設施同時支援 SQL Server 與 PostgreSQL，`./test.sh` 能選擇性啟動兩種 container
4. 引導未來加入 MySQL / Oracle 時，只要**新增 provider 類別 + 註冊工廠**，不需再動抽象層

### 本計畫不涵蓋

- MySQL / MariaDB / Oracle / SQLite 的 provider 實作
- SQL Server → PostgreSQL 資料搬移工具
- 跨 DB 的 ORM 行為差異整理（這是 `Bee.Repository` 層的課題）
- 效能 benchmark 與調校
- PostgreSQL 特有功能（JSONB 查詢、陣列型別、全文搜尋等）— 先做基本 CRUD/DDL，特殊能力留到後續

## 技術設計

### 1. 抽象層萃取

#### 1.1 新增 `ITableSchemaProvider` 介面

於 `src/Bee.Db/Providers/` 新增：

```csharp
public interface ITableSchemaProvider
{
    bool TableExists(string tableName);
    DbTableSchema GetTableSchema(string tableName);
    // 其餘方法依 SqlTableSchemaProvider 公開簽章萃取
}
```

`SqlTableSchemaProvider` 改為 `: ITableSchemaProvider`，不改動行為。

#### 1.2 新增 `ITableRebuildCommandBuilder` 介面

ALTER 計畫引入的 `SqlTableRebuildCommandBuilder` 為 `internal static`，需抽出介面才能讓 orchestrator 走路由。建議簽章：

```csharp
public interface ITableRebuildCommandBuilder
{
    string GetCommandText(TableSchemaDiff diff);
}
```

`SqlTableRebuildCommandBuilder` 從 `internal static` 改為 `internal class : ITableRebuildCommandBuilder`（或保留 `static` 方法並另外包一個實作類）。

#### 1.3 新增 Provider 路由：`DbDialectRegistry` + `IDialectFactory`

**採新 `DbDialectRegistry`**（不擴充 `DbProviderManager`）：後者專責 ADO.NET `DbProviderFactory`，職責單一；`DbDialectRegistry` 專責框架自訂的 builder：

```csharp
public static class DbDialectRegistry
{
    public static void Register(DatabaseType type, IDialectFactory factory);
    public static IDialectFactory Get(DatabaseType type);
}

public interface IDialectFactory
{
    ITableSchemaProvider CreateTableSchemaProvider(string databaseId);
    ICreateTableCommandBuilder CreateCreateTableCommandBuilder();
    ITableAlterCommandBuilder CreateTableAlterCommandBuilder();
    ITableRebuildCommandBuilder CreateTableRebuildCommandBuilder();
    IFormCommandBuilder CreateFormCommandBuilder();
    string GetDefaultValueExpression(FieldDbType dbType);  // 取代 DbFunc.GetSqlDefaultValue
}
```

**相依注入點**：

- `TableSchemaBuilder.CreateComparer()`：改走 `DbDialectRegistry.Get(dbType).CreateTableSchemaProvider(...)`
- `TableUpgradeOrchestrator` 預設建構子：改為 `new TableUpgradeOrchestrator(string databaseId)`，內部查 dialect factory 取得 alter/rebuild/create builder；保留 DI constructor overload 供測試或客製場景使用
- Orchestrator 的 `BuildCreatePlan` / `BuildRebuildPlan` 改用欄位快取的 builder 參考，不再直接 `new` SqlServer 類別

#### 1.4 `DbFunc.GetSqlDefaultValue` 退場

遷移至 `IDialectFactory.GetDefaultValueExpression()`，`DbFunc` 的 `internal` 方法保留薄 wrapper 或直接刪除（`internal` 不影響對外 API）。

### 2. 新增 `DatabaseType.PostgreSQL`

`src/Bee.Definition/DatabaseType.cs` 新增：

```csharp
/// <summary>
/// PostgreSQL.
/// </summary>
PostgreSQL
```

同步在 `DbFunc.DbParameterPrefixes` 與 `DbFunc.QuoteIdentifiers` 各加一筆：

```csharp
{ DatabaseType.PostgreSQL, "@" },           // Npgsql 支援 @name
{ DatabaseType.PostgreSQL, s => $"\"{s.Replace("\"", "\"\"")}\"" }
```

### 3. PostgreSQL Provider 實作

新增目錄 `src/Bee.Db/Providers/PostgreSql/`：

| 檔案 | 職責 |
|------|------|
| `PgDialectFactory.cs` | 實作 `IDialectFactory`，產生下列 builder |
| `PgFormCommandBuilder.cs` | 實作 `IFormCommandBuilder`，產 SELECT/INSERT/UPDATE/DELETE |
| `PgTableSchemaProvider.cs` | 實作 `ITableSchemaProvider`，查 `information_schema.tables`/`information_schema.columns` 與 `pg_catalog`（取得描述用 `pg_description`） |
| `PgCreateTableCommandBuilder.cs` | 實作 `ICreateTableCommandBuilder`，產生 PostgreSQL DDL |
| `PgTableAlterCommandBuilder.cs` | 實作 `ITableAlterCommandBuilder`，處理 `ALTER TABLE ... ADD/ALTER COLUMN`、`CREATE INDEX`、rename（PG：`ALTER TABLE ... RENAME COLUMN`）、comment 同步 |
| `PgTableRebuildCommandBuilder.cs` | 實作 `ITableRebuildCommandBuilder`，產生 tmp 表 + `INSERT SELECT` + rename 序列（PG：`ALTER TABLE ... RENAME TO`） |
| `PgAlterCompatibilityRules.cs`（靜態，internal） | PG 語意的跨型別相容性／narrowing 判定表（對應 SQL Server 版 `SqlAlterCompatibilityRules`） |
| `PgSchemaHelper.cs`（靜態，internal） | PG 識別字 quoting（`"col"`）、`FieldDbType` → PG 型別字串、DEFAULT 表達式（對應 `SqlSchemaHelper`） |
| `PgTypeMapping.cs`（靜態） | `FieldDbType` ↔ PostgreSQL 型別雙向映射表（供 `PgTableSchemaProvider` 讀取 `information_schema` 回映射） |

#### 3.1 PostgreSQL DDL 要點

| 面向 | PostgreSQL 做法 |
|------|----------------|
| 自動流水號 | `GENERATED BY DEFAULT AS IDENTITY`（PostgreSQL 10+，對應 `FieldDbType.AutoIncrement`） |
| UUID 預設值 | `gen_random_uuid()`（需 `pgcrypto` 或 PostgreSQL 13+ 內建）— 或由應用層產 UUID 傳入，避免相依 extension |
| 時間預設值 | `CURRENT_TIMESTAMP`（注意：`now()` 為別名，型別採 `timestamp with time zone`；若要對齊 SQL Server `getdate()` 不含時區語義，改用 `timestamp`） |
| 描述（欄位/表註解） | `COMMENT ON TABLE ... IS '...'` / `COMMENT ON COLUMN ... IS '...'`，取代 `sp_addextendedproperty` |
| 識別字大小寫 | 所有 identifier 一律加 `"` 包裹（維持 PascalCase），避免被自動轉小寫 |
| `NVARCHAR(n)` | 對應 `VARCHAR(n)`（PostgreSQL 無獨立 Unicode 類型） |
| `TEXT` | 對應 `TEXT`（與 SQL Server 相同） |
| `MONEY` / Currency | 建議映射為 `NUMERIC(19,4)`，不用 PostgreSQL 的 `MONEY`（後者有 locale 依賴） |
| `UNIQUEIDENTIFIER` | 對應 `UUID` |
| `VARBINARY(MAX)` / Binary | 對應 `BYTEA` |

#### 3.2 Schema 讀取要點

PostgreSQL 查 `information_schema` 與 `pg_catalog` 混合：

- 表存在性、欄位清單：`information_schema.columns`（跨 DB 標準）
- 欄位描述：`pg_catalog.pg_description`（`information_schema` 不含 comment）
- Primary key / Unique index：`pg_catalog.pg_index` + `pg_class`
- Schema 預設為 `public`，但需支援指定 schema（與 SQL Server 的 `dbo` 對稱）

### 4. 核心設計原則：「能力預備 vs 使用啟用」分層

**現況觀察**：`Bee.Db` 本體**完全不相依任何 ADO.NET driver 套件**，連 `Microsoft.Data.SqlClient` 都沒有。具體啟用哪些 DB，**完全由外部呼叫 `DbProviderManager.RegisterProvider()` 決定**。

| 位置 | 實況 |
|------|------|
| `src/Bee.Db/Bee.Db.csproj` | 僅引 `Bee.Definition`，零 driver 相依 |
| `src/Bee.Db/Providers/SqlServer/*.cs` | **只處理 SQL Server 語法字串生成**，不碰 `SqlConnection` |
| `DbProviderManager` | 存 BCL 內建的 `System.Data.Common.DbProviderFactory`，不綁具體實作 |
| `tests/Bee.Tests.Shared.csproj:17` | Driver 相依寫在**測試端**（`Microsoft.Data.SqlClient` 7.0.0） |
| `tests/Bee.Tests.Shared/GlobalFixture.cs:41` | 測試啟動時才 `DbProviderManager.RegisterProvider(DatabaseType.SQLServer, SqlClientFactory.Instance)` |

**分層設計**：

| 層 | 由誰決定 | 決定了什麼 |
|----|---------|----------|
| `DatabaseType` enum（於 `Bee.Definition`） | 框架列舉 | 本框架**理論上支援的 DB 清單** |
| `DbFunc.QuoteIdentifiers` / `DbParameterPrefixes` | `Bee.Db` 字典 | 每個 DB 的語法常數 |
| `Providers/<Dialect>/*.cs` | `Bee.Db` 類別 | 每個 DB 的語法生成器（純字串，不引 driver） |
| **`DbProviderManager.RegisterProvider()`** | **外部呼叫（應用端／測試端）** | **這次執行實際要啟用哪些 DB 連線** |

這是「**能力預備 vs 使用啟用**」的清楚分層——`Bee.Db` 預備所有可能用到的語法能力，但**不強推任何一個給外部**。外部要啟用哪幾個，呼叫 `RegisterProvider` 時自然決定。`DbFunc.QuoteIdentifiers` 字典已預先放好 Oracle / MySQL / SQLite quoting，也印證此哲學——即使沒人寫對應的 Provider 類別，常數層已預備好。

**PostgreSQL 依此原則處理**（對稱於現有 SQL Server 做法）：

1. **`Providers/PostgreSql/` 直接放進 `Bee.Db`**：與 `Providers/SqlServer/` 對稱，純字串生成類別，不引 Npgsql
2. **`Bee.Db.csproj` 不新增任何 PackageReference**：維持零 driver 相依
3. **`Npgsql` 套件引用寫在測試端**（並由測試端呼叫註冊）：
   ```csharp
   // tests/Bee.Tests.Shared/Bee.Tests.Shared.csproj
   <PackageReference Include="Npgsql" />

   // tests/Bee.Tests.Shared/GlobalFixture.cs 新增
   DbProviderManager.RegisterProvider(DatabaseType.PostgreSQL, NpgsqlFactory.Instance);
   DbDialectRegistry.Register(DatabaseType.PostgreSQL, new PgDialectFactory());
   ```
4. **應用程式端**（使用 Bee.NET 的 ERP）：自行決定引哪個 driver、自行呼叫兩組 `Register()`；`Bee.Db/README.md` 會示範完整註冊範例

#### 4.1 `IDialectFactory` 註冊亦由外部決定（重要）

**設計對稱性**：`DbDialectRegistry` 的使用模式完全對稱於 `DbProviderManager`——**`Bee.Db` 絕不在 `ModuleInitializer` 自動註冊任何 dialect**。

```csharp
// 外部啟動時，兩層註冊都明示
DbProviderManager.RegisterProvider(DatabaseType.SQLServer, SqlClientFactory.Instance);
DbDialectRegistry.Register(DatabaseType.SQLServer, new SqlDialectFactory());

DbProviderManager.RegisterProvider(DatabaseType.PostgreSQL, NpgsqlFactory.Instance);
DbDialectRegistry.Register(DatabaseType.PostgreSQL, new PgDialectFactory());
```

這樣保留三個價值：
- **一致性**：兩層註冊同一個模式，認知負擔低
- **彈性**：使用者可替換內建 dialect（例如針對公司自訂的 SQL Server 變體）
- **純粹性**：`Bee.Db` 不做任何「幫你做決定」的事

**使用者體驗緩解**：若覺得「兩組 Register 太囉嗦」，可在未來提供便利方法（不在本計畫範圍）：
```csharp
// 可能的 DX 糖衣，留待未來考慮
BeeDb.EnableSqlServer(SqlClientFactory.Instance);   // 一鍵註冊 provider + dialect
BeeDb.EnablePostgreSQL(NpgsqlFactory.Instance);
```
但底層 API 仍是兩組 `Register`，糖衣只是 convenience wrapper。

此策略好處：
- **零新套件**：不發佈 `Bee.Db.PostgreSql`，版本管理不增複雜度
- **使用者只付需要的代價**：引哪個 driver 就吃哪份套件成本
- **未來加 MySQL / Oracle 時同模式**：直接擴充 `Providers/MySql/`、`Providers/Oracle/`，零決策成本
- **對稱性清晰**：`Bee.Db` 的 `Providers/` 目錄 = 「框架備好的語法生成器清單」；實際啟用由外部決定

### 5. 連線字串範本

`DatabaseItem.ConnectionString` 目前支援 `{@DbName}`、`{@UserId}`、`{@Password}` placeholder。PostgreSQL 的連線字串格式：

```
Host=localhost;Port=5432;Database={@DbName};Username={@UserId};Password={@Password}
```

placeholder 機制 provider-agnostic，無需改動。

## 測試策略

### 多 DB 並行策略：每 DB 一條環境變數 + 參數化 attribute

採可擴展設計，未來新增 MySQL / Oracle 等不需修改基礎設施。

#### 環境變數命名

`BEE_TEST_CONNSTR_{DBTYPE}`，`{DBTYPE}` 為 `DatabaseType` enum 值的 uppercase。

| 變數 | 對應 DB |
|------|--------|
| `BEE_TEST_CONNSTR_SQLSERVER` | SQL Server |
| `BEE_TEST_CONNSTR_POSTGRESQL` | PostgreSQL |
| 未來 `BEE_TEST_CONNSTR_MYSQL` | MySQL |
| 未來 `BEE_TEST_CONNSTR_ORACLE` | Oracle |

舊 `BEE_TEST_DB_CONNSTR` 一次性遷移至 `BEE_TEST_CONNSTR_SQLSERVER`，不保留 alias（內部 infra，無外部消費者）。

#### 連線 ID 命名

`common_{dbtype_lower}` —— 與 DB 類型 1:1 對應：

- `common_sqlserver`
- `common_postgresql`

`DbGlobalFixture` 對每個有 connection string 的 DB 各自註冊一個 `DatabaseItem`。

#### `[DbFact]` / `[DbTheory]` 強制參數化

```csharp
[DbFact(DatabaseType.PostgreSQL)]
public void PgFoo() { ... }

[DbFact(DatabaseType.SQLServer)]
public void SqlFoo() { ... }
```

實作邏輯：依 `DatabaseType` 推導 env var 名稱，未設則 Skip。新增 DB 不需新 attribute 類別。

#### `DbGlobalFixture` 多 DB 並存（容錯）

針對每個 `DatabaseType`：

- 偵測對應 env var；未設則跳過該 DB
- 連線失敗只跳過該 DB（記 warning + 不 throw），不阻擋其他 DB 的 fixture 進行
- 執行 schema 建立（`TableSchemaBuilder.Execute(connId, "st_user")` / `st_session`）
- 執行 seed：依方言產 INSERT（PG：`gen_random_uuid()` / `CURRENT_TIMESTAMP`；SQL：`NEWID()` / `GETDATE()`）

採此設計後，本機只啟動單一 DB 容器即可跑該 DB 的所有 `[DbFact]` 測試；CI 啟動兩個 service container 則同時跑兩 DB。

#### `./test.sh` 擴充

```bash
SQL_CONTAINER="${BEE_TEST_SQL_CONTAINER:-sql2025}"
PG_CONTAINER="${BEE_TEST_PG_CONTAINER:-pgvector-db}"

start_container "$SQL_CONTAINER" 1433
start_container "$PG_CONTAINER"  5432
```

兩個容器各自試啟，未存在/未啟動就跳過該 DB 測試。

#### 環境變數對照

## 風險與未決事項

### 已知風險

1. **PostgreSQL 識別字大小寫在 `TableSchemaComparer` 比對階段可能 false positive**
   - FormSchema 的 `Employee` 表名在 PostgreSQL 實際為 `"Employee"`，但比對時 `information_schema.columns.table_name` 回傳原樣（含大小寫），應正確；需在 PR 6 驗證
2. **`gen_random_uuid()` 依賴 PostgreSQL 13+ 或 `pgcrypto`**
   - 緩解：明文文件要求 PostgreSQL 13+，或應用層產 UUID 傳入，避免 DDL 相依 extension
3. **`timestamp` vs `timestamp with time zone` 語義選擇**
   - SQL Server `datetime` 不含時區，對齊選 `timestamp`；若未來要引入時區感知，另案處理
4. **`Npgsql` 在 `Bee.Tests.Shared` 引入時的 analyzer 警告**（`TreatWarningsAsErrors=true`）
   - 緩解：PR 6 引入 Npgsql 時先做 smoke build，若有警告逐條處理，不以 `#pragma disable` 壓制

### 未決事項（待使用者確認）

1. ~~**`[PgDbFact]` 命名 vs 改造 `[DbFact]` 接受目標 DB 參數**~~
   - ✅ 已解：採參數化 `[DbFact(DatabaseType)]`（單一 attribute、可擴展），詳見 §測試策略
2. **是否同步新增 `Providers/MySql/` 骨架**？
   - 建議：不同步；先把 PostgreSQL 做穩，MySQL 留到下一個迭代
3. ~~**現有 `plan-alter-based-upgrade.md`（Schema 升級計畫）與本計畫的順序關係**~~
   - ✅ 已解：ALTER-based 升級計畫已於 2026-04-25 完成（v4.1.0）；本計畫 PR 1 改為雙層（1a 抽象、1b 路由接入）以覆蓋 orchestrator 帶進來的新耦合點
4. **發佈版號**：發佈先暫緩，待後續確認再決定版號。原規劃為 PR 1a + 1b 後發 `v4.2.0`（純重構），PostgreSQL 完成後發 `v4.3.0`（新增能力）。
5. ~~**`ITableRebuildCommandBuilder` 命名與可見度**~~
   - ✅ 已解：採 `public interface` + `internal class` 慣例（已於 PR 1a 落地，commit `3df46bd`）。

### 已知測試 side-effect（與本計畫無關，留待另案處理）

1. **`tests/Define/DbSchemaSettings.xml` 被 `LocalDefineAccessSaveTests.SaveDbSchemaSettings_*` 系列測試覆寫**：每次跑完測試會重序列化此檔案，造成 `<Tables>` 元素遺失。需 `git checkout -- tests/Define/DbSchemaSettings.xml` 還原。修法應是該測試使用獨立 tmp 目錄而非污染 fixture 檔案。
2. **`Bee.Api.Client.UnitTests.ApiConnectValidatorTests.Validate_ValidUrl_ReturnsRemoteConnectType`** 為 `[LocalOnlyTheory]`，需要本機跑著 API server 才會通過。CI 環境會 skip（`CI=true`）。
